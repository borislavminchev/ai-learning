## Step-by-step

### Step 1 â€” VRAM estimator (predict concurrency before you rent)

**What.** Write `vram_estimate.py` that pulls a model's `config.json` from the Hub (or a pasted dict), and prints: **weights GB**, **KV cache GB per 1k tokens per sequence**, and **max concurrent sequences** at a target context length for a target card size. Validate against your Week-1 vLLM startup log.

**Why.** This is the single most valuable hour of the phase (lecture 06). Weights decide *whether the model fits*; the **KV cache decides how many users it serves**, and KV is what OOMs vLLM at startup when `--max-num-seqs Ã— --max-model-len` implies more blocks than fit. The formula:

```
weights_GB       â‰ˆ params_B Ã— bytes_per_param            # fp16=2, fp8=1, int4â‰ˆ0.5
kv_bytes/token   = 2 Ã— n_layers Ã— n_kv_heads Ã— head_dim Ã— kv_bytes   # K and V; use num_KEY_VALUE_heads!
head_dim         = hidden_size / num_attention_heads
kv_pool_avail_GB = card_GB Ã— gpu_mem_util âˆ’ weights_GB âˆ’ headroom_GB  # headroom â‰ˆ 1.5 GB
max_num_seqs     â‰ˆ kv_pool_avail_GB / (kv_bytes/token Ã— context_len)
```

The classic error is using `num_attention_heads` instead of `num_key_value_heads` in the KV formula â€” on a GQA model that over-estimates KV by 4â€“8Ã—.

**Do it.** Sketch (fill in the arithmetic yourself â€” don't just paste):

```python
# vram_estimate.py
from dataclasses import dataclass
from huggingface_hub import hf_hub_download
import json, argparse

BYTES_PER_PARAM = {"fp16": 2, "bf16": 2, "fp8": 1, "int4": 0.5}

def load_config(model_id: str) -> dict:
    path = hf_hub_download(model_id, filename="config.json")  # needs HF_TOKEN for gated
    return json.loads(open(path).read())

def kv_bytes_per_token(cfg: dict, kv_bytes: int = 2) -> int:
    n_layers  = cfg["num_hidden_layers"]
    n_kv      = cfg["num_key_value_heads"]              # GQA number â€” the one that matters
    head_dim  = cfg["hidden_size"] // cfg["num_attention_heads"]
    return 2 * n_layers * n_kv * head_dim * kv_bytes    # 2 = K and V

def estimate(cfg, params_b, card_gb, ctx_len,
             weight_dtype="fp16", util=0.90, headroom_gb=1.5, kv_bytes=2):
    weights_gb = params_b * BYTES_PER_PARAM[weight_dtype]
    kv_tok     = kv_bytes_per_token(cfg, kv_bytes)      # bytes/token
    kv_per_seq_gb = kv_tok * ctx_len / 1e9
    kv_pool_gb    = card_gb * util - weights_gb - headroom_gb
    max_seqs      = max(0, kv_pool_gb / kv_per_seq_gb)
    return {
        "weights_gb": round(weights_gb, 2),
        "kv_gb_per_1k_per_seq": round(kv_tok * 1000 / 1e9, 4),
        "kv_pool_gb": round(kv_pool_gb, 2),
        "max_num_seqs": int(max_seqs),
    }

if __name__ == "__main__":
    ap = argparse.ArgumentParser()
    ap.add_argument("--model", required=True)
    ap.add_argument("--params-b", type=float, required=True)  # e.g. 7.6 for Qwen2.5-7B
    ap.add_argument("--card-gb", type=float, default=24)
    ap.add_argument("--ctx", type=int, default=8192)
    ap.add_argument("--weight-dtype", default="fp16")
    ap.add_argument("--kv-bytes", type=int, default=2)        # use 1 for --kv-cache-dtype fp8
    args = ap.parse_args()
    cfg = load_config(args.model)
    print(estimate(cfg, args.params_b, args.card_gb, args.ctx,
                   args.weight_dtype, kv_bytes=args.kv_bytes))
```


**Expected result.** For a 7B-class GQA model at 8k on 24 GB you'll see roughly `weights_gb â‰ˆ 14â€“16`, `kv_gb_per_1k_per_seq â‰ˆ 0.1â€“0.13`, and `max_num_seqs` in the low-to-mid teens (the exact value depends on the config and your dtype). A Llama-3-8B-shaped config gives ~128 KB/token â†’ ~1 GB/seq at 8k â†’ single-digit concurrency on 24 GB at fp16 (lecture 06 worked example).

**Verify.** Compare against your Week-1 vLLM **startup log**. Modern vLLM prints one of:
```

# or:

```
Convert blocks to a concurrency check (block size â‰ˆ 16 tokens):
```

seqs_at_ctx  = tokens_of_kv / context_len
```
**Definition-of-Done gate:** your predicted `max_num_seqs` must be **within ~20%** of what vLLM reported. If not, the usual culprits are: you used `num_attention_heads` (over-estimate), you forgot KV stays fp16 even when weights are int4 (under-estimate), or your `headroom_gb` guess is off â€” tighten it and reconverge.

**Troubleshoot.**

### Step 3 â€” Load test with k6 (tokens/sec, TTFT, TPOT at p50/p95/p99)

**What.** Run a **closed-loop** (`constant-vus`) k6 test at concurrency **1 / 8 / 64**, each 60 s, with a **pinned workload**. Add a **streaming** variant to split TTFT from TPOT. Write p50/p95/p99 for each tier to `results/latency.csv`.

**Why.** Concurrency trades throughput against tail latency: tokens/sec climbs while p99 TTFT explodes past the "knee" (lecture 08). You must report both, at a stated concurrency, or your numbers are marketing. Pin input tokens, `max_tokens`, and `temperature=0` so runs are comparable, and discard a warm-up window so you measure steady state, not CUDA-graph cold start.

**Do it (basic â€” status/throughput).** `loadtest/script.js`:

```javascript
import http from 'k6/http';
import { check } from 'k6';

const BASE = __ENV.BASE_URL || 'http://localhost:8000/v1';
const MODEL = __ENV.MODEL || 'Qwen/Qwen2.5-7B-Instruct';

export const options = {
  discardResponseBodies: false,
  scenarios: {
    c1:  { executor: 'constant-vus', vus: 1,  duration: '60s', startTime: '0s'   },
    c8:  { executor: 'constant-vus', vus: 8,  duration: '60s', startTime: '75s'  },
    c64: { executor: 'constant-vus', vus: 64, duration: '60s', startTime: '150s' },
  },
  // discard a warm-up window: k6 has no built-in skip, so the first tier
  // doubles as warm-up â€” ignore c1's first ~10s when reading results.
};

const BODY = JSON.stringify({
  model: MODEL,
  messages: [{ role: 'user', content: 'Write exactly a 100-word product blurb about a smart water bottle.' }],
  max_tokens: 150, temperature: 0,          // pinned for comparability
});

export default function () {
  const res = http.post(`${BASE}/chat/completions`, BODY,
    { headers: { 'Content-Type': 'application/json' } });
  check(res, { 'status 200': (r) => r.status === 200 });
}
```

Run it (generator **in-region / on the GPU box** â€” never from your laptop over the internet, or network latency swamps TTFT):
```bash

```

**Do it (streaming â€” true TTFT vs TPOT).** k6's default HTTP client returns only when the response completes, so it can't time the first SSE chunk. Use k6's experimental streaming client (or the locust variant below). Signatures/shape:

```javascript
// loadtest/stream.js  â€” TTFT via first SSE byte
import { Client } from 'k6/experimental/streams';   // if unavailable on your k6, use locust
import { Trend } from 'k6/metrics';
const ttft = new Trend('ttft_ms', true);
const tpot = new Trend('tpot_ms', true);
// body must set  "stream": true
// on first data chunk:  ttft.add(now - sent)
// at stream end:        tpot.add((now - firstChunk) / (tokens - 1))
```

**Locust alternative (Python, easiest for streaming).** `loadtest/locustfile.py`:
```python
import time, json
from locust import HttpUser, task, constant

class LLMUser(HttpUser):
    wait_time = constant(0)          # closed-loop: fire next as soon as prev finishes
    @task
    def chat(self):
        body = {"model": "Qwen/Qwen2.5-7B-Instruct",
                "messages": [{"role": "user", "content": "100-word blurb about a smart water bottle."}],
                "max_tokens": 150, "temperature": 0, "stream": True}
        sent = time.time(); first = None; ntok = 0
        with self.client.post("/v1/chat/completions", json=body, stream=True,
                              catch_response=True) as r:
            for line in r.iter_lines():
                if not line or not line.startswith(b"data:"): continue
                if line.strip() == b"data: [DONE]": break
                if first is None:
                    first = time.time()
                    self.environment.events.request.fire(  # log TTFT as a custom metric
                        request_type="TTFT", name="ttft", response_time=(first-sent)*1000,
                        response_length=0)
                ntok += 1
            if first and ntok > 1:
                tpot = (time.time() - first) / (ntok - 1) * 1000
                self.environment.events.request.fire(
                    request_type="TPOT", name="tpot", response_time=tpot, response_length=0)
```
Run at each tier:
```bash
uv run locust -f loadtest/locustfile.py --host "$(dirname $BASE_URL)" \
     --headless -u 8 -r 8 -t 60s --csv results/locust_c8
```
(Repeat with `-u 1` and `-u 64`; `-u` = users = concurrency, `-r` = ramp rate.)

**Compute TTFT/TPOT correctly** (lecture 08):
```
TTFT = t(first SSE chunk) âˆ’ t(request sent)
TPOT = (t(last chunk) âˆ’ t(first chunk)) / (output_tokens âˆ’ 1)     # inter-token gap
E2E  = TTFT + (output_tokens âˆ’ 1) Ã— TPOT
```

Record into `results/latency.csv` with columns:
```

```

**Expected result.** Throughput scales steeply (roughly 1 â†’ 8 â†’ 64 gives ~25Ã— tokens/sec by conc 64), TPOT stays nearly flat at low concurrency then degrades, and **TTFT p99 explodes** at conc 64 (queueing). Shape from lecture 08's representative table: conc 1 â‰ˆ 25 tok/s / TTFT ~250 ms; conc 8 â‰ˆ 180 tok/s / TTFT p95 ~520 ms; conc 64 â‰ˆ 620 tok/s / TTFT p99 ~7 s. Your absolute numbers depend on the card and model â€” the *shape* is the lesson.

**Verify.**
- While each tier runs, watch `nvitop`: high memory + **low SM util during decode** = memory-bound (expected). Low memory + low util = you're wasting the card (raise `--max-num-seqs`).
- Cross-check k6/locust TTFT against `vllm:time_to_first_token_seconds` and `vllm:request_queue_time_seconds` from `/metrics`. If client TTFT p99 is 7 s but server queue-time p99 is 6.5 s, the tail is **queueing** (fix: fewer VUs or add replicas). If server TTFT is 300 ms but client sees 3 s, the gap is **network** (run the generator on-box).

**Troubleshoot.**
- k6 streaming client unavailable in your version â†’ use the **locust** variant for TTFT/TPOT and keep k6 only for throughput/status.
- p99 dominated by the internet â†’ move the generator onto the pod (`ssh` in and run k6 there) or use `localhost`.
- Numbers not comparable between runs â†’ you didn't pin `max_tokens`/`temperature`, or you didn't discard warm-up. Re-pin and drop the first ~10â€“15 s.
- Coordinated omission worry â†’ closed-loop (`constant-vus`) *understates* overload tails by design. To find your true breaking RPS, add an **open-loop** scenario (`executor: 'constant-arrival-rate'` with `preAllocatedVUs`) â€” see stretch goals.

### Step 4 â€” Cost per 1M tokens and the break-even chart

**What.** In `results/breakeven.py`, turn your best-tier sustained tokens/sec into `cost_per_1M`, then plot **monthly cost vs monthly token volume** for (a) dedicated GPU 24/7 (flat, stepping when it saturates), (b) hosted API (linear, priced on **both** input and output at your real ratio), and (c) serverless scale-to-zero (low slope + small cold-start term). Mark the crossover; save `breakeven.png`.

**Why.** A flat line (dedicated `$/hr`) and a rising line (API `$/token`) cross exactly once. Below it, call the API; above it, the box wins (lecture 09). The honesty comes from three details most charts skip: price **both** token directions at your measured ratio, **step** the dedicated line with `ceil()` when you outgrow one GPU, and carry a **utilization** correction + **ops-tax** fudge so you report the honest crossover, not the 100%-utilization fantasy.

The core formulas:
```
cost_per_1M_tokens = (1e6 / sustained_tok_per_sec / 3600) Ã— gpu_$/hr
crossover_V        = (730 Ã— gpu_$/hr) / blended_$_per_1M          # tokens/month (millions)
effective_cost/1M  = cost_per_1M / utilization                    # spiky box costs more per token
```

**Do it.** `results/breakeven.py` (extend the lecture-09 skeleton; plug in *your* measured `T` and rate):

```python
import numpy as np, matplotlib.pyplot as plt

# --- your measured / chosen inputs ---
T_tok_s   = 620          # sustained tokens/sec at your best tier (from latency.csv)
gpu_hr    = 0.75         # $/hr for your card
ops_tax   = 1.4          # +40% for on-call/upgrades/idle firefighting (lecture 09)
util      = 0.40         # realistic duty cycle (spiky daytime traffic)
in_frac, out_frac = 5/6, 1/6          # your measured 5:1 input:output ratio
p_in, p_out = 0.20e-6, 0.20e-6        # hosted open-model API $/token (both directions)

V = np.linspace(0, 3e9, 500)          # tokens/month, 0..3B

cost_per_1M = (1e6 / T_tok_s / 3600) * gpu_hr
print(f"cost/1M @100% util = ${cost_per_1M:.3f}; effective @{util:.0%} = ${cost_per_1M/util:.3f}")

api = V*in_frac*p_in + V*out_frac*p_out
box_tokens_mo = T_tok_s * 3600 * 730                       # capacity of one box at 100% util
n_boxes   = np.ceil(np.maximum(V, 1) / box_tokens_mo)       # STEP when saturated
dedicated = n_boxes * 730 * gpu_hr * ops_tax
serverless = V * (gpu_hr / 3600 / T_tok_s) + 5             # ~$/token active + tiny warm floor

cross = (730 * gpu_hr * ops_tax) / ((in_frac*p_in + out_frac*p_out) * 1e6)  # in millions
plt.plot(V, api, label="hosted API")
plt.plot(V, dedicated, label="dedicated 24/7 (+ops tax)")
plt.plot(V, serverless, label="serverless scale-to-zero")
plt.axvline(cross*1e6, ls="--", label=f"crossover â‰ˆ {cross:.0f}M tok/mo")
plt.xlabel("tokens / month"); plt.ylabel("$ / month"); plt.legend()
plt.savefig("results/breakeven.png", dpi=120, bbox_inches="tight")
```

### Step 5 â€” Write the recommendation

## Troubleshooting cheatsheet

| Symptom | Likely cause | Fix |
|---|---|---|
| Estimator off by 4â€“8Ã— (too high) | Used `num_attention_heads` in KV formula | Use `num_key_value_heads` (the GQA number) |
| Estimator too low vs vLLM | Assumed int4 KV because weights are int4 | KV stays fp16 (2 bytes) unless `--kv-cache-dtype fp8`; set `--kv-bytes 2` |
| Estimator off by ~15â€“20% | `headroom_gb` guess | Tighten headroom toward ~1â€“1.5 GB and reconverge |
| TTFT p99 huge but throughput fine | Queueing at high concurrency (past the knee) | Cross-check `vllm:request_queue_time_seconds`; drop to a lower tier or add replicas |
| Client TTFT â‰« server TTFT | Network latency (laptop over internet) | Run the generator on the GPU box / same VPC; use `localhost` |
| Runs not comparable | Unpinned `max_tokens`/`temperature`, or no warm-up discard | Pin `temperature:0`, fixed `max_tokens`; drop first ~10â€“15 s |
| k6 can't measure TTFT | Non-streaming HTTP client | Use `stream:true` + streaming client, or the locust variant |
| Break-even crossover absurd | Priced only output, or frontier vs open-model mismatch | Price both directions at measured ratio; compare like-for-like |
